# Accessibility (A11y) Guide

Curatarr targets WCAG 2.1 Level AA compliance. This document is the minimal
reference for developers and agents making UI changes.

## Quick Rules

| Rule | Pattern |
|------|---------|
| Buttons/links with visible text need no `aria-label` | The visible text IS the accessible name |
| Icon-only buttons/links must have `aria-label` | Never rely on `title` alone |
| Chip remove buttons (e.g. "Drama ×") need augmented `aria-label` | `aria-label="Remove Drama filter"` |
| Inputs/selects without a visible `<label>` must have `aria-label` | Inline the string — it's i18n content |
| Disclosure buttons (dropdowns) must reflect state | `aria-expanded` + `aria-haspopup` — no `aria-label` needed if button has text |
| Progress indicators must use ARIA roles | `role="progressbar"` + `aria-valuenow/min/max` |
| Charts (SVG) must have a text alternative | Wrap with `role="img"` + `aria-label` |
| `aria-label` strings belong in i18n, not a parallel module | Inline short strings; extract to namespaces when i18n is implemented |

---

## When to use `aria-label`

**Do NOT add `aria-label` to a button that already has visible text.** The accessible name is computed from content — adding `aria-label` overrides it, which breaks screen readers when labels drift from the visible copy.

**DO add `aria-label` for:**

1. **Icon-only controls** — no visible text at all
2. **Ambiguous remove/close chips** — e.g. a button showing `Drama ×` where `×` alone is meaningless; use `aria-label="Remove Drama filter"`
3. **Unlabelled form controls** — `<input>` or `<select>` with no associated `<label>` element
4. **Structural ARIA roles** — progressbar, img wrappers, etc.

**Do NOT add `aria-label` for:**
- Buttons that have readable text content
- Disclosure buttons (use `aria-expanded`/`aria-haspopup` instead)
- `<select>` whose first `<option>` already serves as a placeholder (e.g. "Select existing tag")

---

## i18n note

Accessible label strings are UI copy. When react-i18next is fully adopted, they will be extracted into namespace JSON files like any other user-visible string. Do not create a parallel `a11y/labels.ts` module — that creates duplicate content outside the i18n pipeline.

For now, inline short strings directly. Keep them consistent with visible copy where the control has both.

## Common Patterns

### Icon-only button
```tsx
<button type="button" aria-label="Close">
  <X size={18} />
</button>
```

### Icon-only link
```tsx
<Link to={href} aria-label="Open full movie page">
  <ExternalLink size={13} />
</Link>
```

### Disclosure button (dropdown) — no `aria-label` needed
```tsx
<button
  type="button"
  aria-expanded={isOpen}
  aria-haspopup="listbox"
>
  {selectedGenres.length > 0 ? `${selectedGenres.length} selected` : 'Select genres'}
</button>
```

### Chip remove button — augmented label
```tsx
<button
  type="button"
  aria-label={`Remove ${genre} filter`}
>
  {genre} ×
</button>
```

### Form input without visible label
```tsx
<input
  type="text"
  aria-label="Search titles"
  placeholder="Search titles…"
/>
```

### Select without visible label
```tsx
<select aria-label="Audio format filter">
  <option value="">Format</option>
  …
</select>
```

### Progress bar
```tsx
<div
  role="progressbar"
  aria-valuenow={pct}
  aria-valuemin={0}
  aria-valuemax={100}
  aria-label="Library scan progress"
  style={{ width: `${pct}%` }}
/>
```

### Chart (SVG)
```tsx
<div role="img" aria-label="Resolution distribution">
  <ResponsiveContainer …>…</ResponsiveContainer>
</div>
```

## Automated Checks

The e2e suite runs axe on every route: `npm run test:e2e`

File: `test/a11y.spec.cjs`

The spec scans each route with `wcag2a` and `wcag2aa` tags and attaches a JSON
report per route as a Playwright test attachment. It does not fail the run on
violations — violations are captured for review. Run with `PW_E2E_WORKERS=1` on
local dev (the default).

To review violations from a failed run, check `test/results/` for the JSON
attachments before they are cleaned up.

## Agent Checklist

Before marking any UI task `DONE`, confirm:

- [ ] Every new button/link with no visible text has an inline `aria-label`
- [ ] Buttons with visible text do NOT have a redundant `aria-label`
- [ ] Icon-in-chip remove buttons use `aria-label="Remove <item> filter"` pattern
- [ ] Every new `<input>`/`<select>` without a visible `<label>` has an inline `aria-label`
- [ ] Every disclosure button that opens a dropdown has `aria-expanded` + `aria-haspopup`
- [ ] Every new progress indicator has `role="progressbar"` + `aria-valuenow/min/max`
- [ ] Every new chart has a wrapping `role="img"` element with `aria-label`
- [ ] No new `a11y/labels.ts` or parallel a11y string module has been introduced
